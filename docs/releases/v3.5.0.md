# Locus v3.5.0 Release Notes

**Release focus:** one-command Codex install.

`v3.5.0` packages Locus as the public `locus-memory` npm runtime and adds a Codex installer so new users no longer need to clone the repository and hand-wire `dist/server.js` for the normal path.

## Highlights

- `npx -y locus-memory@latest install codex`
- `npx -y locus-memory@latest doctor codex`
- `npx -y locus-memory@latest uninstall codex --yes`
- generated Codex marketplace bundle under `dist/marketplace/`
- package contract for `dist/server.js`, `dist/cli.js`, and the canonical Codex skill
- manual MCP fallback remains documented and supported

## What The Installer Does

The Codex installer:

- installs the canonical Locus skill into `$CODEX_HOME/skills/locus-memory/SKILL.md`
- configures the `locus` MCP server
- uses `redacted` capture defaults for practical recall
- writes a recurring runtime command pinned to `locus-memory@3.5.0`
- avoids writing `@latest` into recurring MCP config
- uses an install lock and safe temp-file writes
- detects existing manual Locus MCP entries and foreign `locus` entries separately

## Validation Summary

Validation covered:

- package contract tests
- CLI command tests
- Codex install, doctor, uninstall, lock, cleanup, and config tests
- local tarball runtime smoke
- generated marketplace bundle tests
- `npm publish` prepublish gate: typecheck, Biome lint, `105` test files / `1098` tests, and build
- npm registry metadata for `locus-memory@3.5.0`
- registry-hosted `npx -y locus-memory@3.5.0 --version`
- registry-hosted `npx -y locus-memory@3.5.0 --help`
- disposable `CODEX_HOME` install from npm
- local Codex config migration to the package runtime

The validated recurring MCP command is pinned:

```toml
[mcp_servers.locus]
command = "npx.cmd"
args = ["-y", "locus-memory@3.5.0", "mcp"]
```

## Known Limitations

- Codex CLI is the validated primary path.
- Codex desktop / extension parity remains unverified until tested in that surface.
- First run of the package runtime may need network access if the npm cache does not already contain the pinned package.
- Migrated manual installs can report `Ownership: missing` in `doctor codex`; this does not block runtime use, but a future cleanup can add an explicit ownership-claim migration.
- If a future package validation failure appears after publication, prefer a patch release and documented workaround over rewriting history.

## Upgrade Notes

Existing manual MCP users can keep their setup. To migrate to the package-owned runtime:

```bash
npx -y locus-memory@latest install codex
npx -y locus-memory@latest doctor codex
```

If you need to stay on a local checkout, keep using the manual MCP fallback:

```bash
codex mcp add locus -- node /path/to/locus/dist/server.js
```
