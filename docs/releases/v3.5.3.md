# Locus v3.5.3 Release Notes

**Release focus:** one-command Codex install with verified npm runtime.

`v3.5.3` is the public Track B release for Codex onboarding. It packages Locus as the `locus-memory` npm runtime, adds a Codex installer/doctor/uninstall flow, keeps manual MCP setup as a fallback, and includes the Windows/Codex hotfixes found during local post-publish validation.

## Install

```bash
npx -y locus-memory@latest install codex
```

Then restart Codex and verify:

```bash
npx -y locus-memory@latest doctor codex
```

## What Shipped

- Public npm package: `locus-memory@3.5.3`
- CLI entrypoint: `locus-memory`
- MCP runtime command: `locus-memory mcp`
- Codex installer: `locus-memory install codex`
- Codex diagnostics: `locus-memory doctor codex`
- Codex removal helper: `locus-memory uninstall codex --yes`
- Generated marketplace bundle under `dist/marketplace/`
- Canonical Codex skill installed to `$CODEX_HOME/skills/locus-memory/SKILL.md`
- Safe `redacted` capture defaults for practical Codex recall
- Version-pinned recurring MCP runtime instead of `@latest`
- GitHub Pages landing page with copyable Codex install command

## Important Fixes

### Safe MCP Working Directory

`v3.5.1` fixed a Windows startup failure where Codex could launch the recurring MCP command from inside the Locus monorepo:

```toml
args = ["-y", "locus-memory@3.5.1", "mcp"]
```

When the current project was also named `locus-memory`, `npx` could resolve local workspace context instead of the registry package and close before MCP `initialize`.

The installer now writes:

```toml
[mcp_servers.locus]
command = "npx.cmd"
args = ["-y", "locus-memory@3.5.3", "mcp"]
cwd = "C:\\Users\\<you>\\.codex"
```

On non-Windows systems the same idea applies: the recurring MCP process starts from `$CODEX_HOME`, not from the active project checkout.

### Accurate Doctor Ownership

`v3.5.2` fixed `doctor codex` reporting:

```text
Ownership: missing
```

for real package-owned installs. The command now falls back to:

```text
codex mcp get locus
```

and classifies the real Codex MCP entry as `package-owned`, `manual-locus`, `foreign-locus`, or `missing`.

## Validation

Local and registry-backed release validation completed before GitHub publication:

- `npm run check`: typecheck, lint, and `105` test files / `1101` tests passed
- `npm run build`: regenerated `dist/cli.js` and `dist/server.js`
- `npm pack --dry-run --json --cache .npm-cache`: package contains the expected 7 runtime files
- `npm view locus-memory@3.5.3`: registry metadata, tarball, integrity, and `bin` confirmed
- `npx -y locus-memory@3.5.3 --version` from `$CODEX_HOME`: returned `3.5.3`
- `npx -y locus-memory@3.5.3 install codex --yes`: updated local Codex config
- `codex mcp get locus`: showed `args: -y locus-memory@3.5.3 mcp` and `cwd: C:\Users\Admin\.codex`
- `npx -y locus-memory@3.5.3 doctor codex`: reported `Ownership: package-owned`
- raw MCP `initialize` through published `npx -y locus-memory@3.5.3 mcp`: returned `protocolVersion: 2024-11-05`

Developer note: running `npx -y locus-memory@3.5.3 --version` from inside the local Locus monorepo can still hit npm's workspace-name edge case on Windows. The installed recurring MCP runtime starts from `$CODEX_HOME`, and that path is validated above.

## What This Means For Users

New Codex users no longer need to clone the repository, build `dist/server.js`, and hand-edit MCP configuration for the normal path. The npm installer configures the skill, MCP server, safe capture defaults, and recurring runtime command.

Manual MCP setup remains supported for development checkouts and fallback troubleshooting.

## Known Boundaries

- Codex CLI is the primary validated path.
- Codex desktop / extension uses the same MCP configuration model where exposed by the upstream surface, but parity remains unverified until tested directly there.
- `redacted` is the recommended practical recall mode. `full` is explicit warning territory.
